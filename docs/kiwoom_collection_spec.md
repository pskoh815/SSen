# 키움 OpenAPI+ 거래대금 수집 — 검증 결과 및 운영 스펙

`docs/api_collection.md`의 data.go.kr 경로를 대체하는 신규 수집 경로.
**data.go.kr은 폴백으로 유지** — 키움 로그인 실패/장애 시 자동 전환할 수 있도록 코드는 보존.

## 데이터 소스별 역할 정리 (2026-06-17, OHLCV까지 4종 통합 완료)

| 데이터 | 키움 TR | 키움 커버리지 | data.go.kr 역할 |
|---|---|---|---|
| 거래대금 | OPT10032 | 당일만 (실시간 조회) | 실패 시 폴백 + 과거 백필 |
| 코스피시세 | OPT20006 | **당일+과거 백필 모두** (차트형 TR) | 실패 시 폴백만 (백필도 키움 우선) |
| ADR(상승하락비율) | OPT20009 | 당일만 (과거 조회 TR이 키움에 없음) | **과거 백필 전용** (폴백 아닌 역할 분리) |
| 개별종목 OHLCV | OPT10081 | **universe(거래대금 상위 1년 누적) 한정 당일+백필** | universe 밖 종목 + 실패 시 폴백 (전종목 커버) |

거래대금/ADR은 "당일 실시간 조회 TR"이라 과거 날짜를 요청해도 데이터가 없어 키움으로
백필이 불가능하지만, 코스피시세(OPT20006)는 OPT10081(개별종목 일봉)과 동일한 "차트형 TR"이라
임의의 과거 `기준일자`를 줘도 그 시점 기준 수백 거래일치를 한 번에 반환한다 — 이 차이가
세 데이터의 daily_update.py 통합 구조(거래대금/ADR은 키움-실패시폴백 vs 코스피시세는
키움이 폴백 대상이 아니라 사실상 유일경로)를 결정한 핵심 근거.

## 검증된 사실 (2026-06-17 실측, 1차 출처)

| 항목 | 결과 |
|---|---|
| 수집 가능 시점 | 장 마감(15:30) 후 **15:40~16:00** — data.go.kr(익일 09:00+) 대비 약 17시간 단축 |
| 거래대금 단위 | **백만원** — 삼성전자 실측값 6,085,609 × 1,000,000 ≈ 6.09조원(실제와 일치) |
| 거래량 단위 | 주 (data.go.kr과 동일, 변환 불필요) |
| 가격/대비 컬럼 형식 | `+346500`, `-16000` 같은 부호 포함 문자열 — `float()` 캐스팅 시 부호 자동 처리됨 |
| 전일대비기호 | 1=상한 2=상승 3=보합 4=하한 5=하락 |
| ETF/ETN 혼입률 | 거래대금 상위 100종목 중 **약 41%** (KODEX/TIGER/SOL 등 신종 레버리지 ETF 다수) → 반드시 제외 |
| 환경 제약 | 32비트 Python 필수 (키움 OpenAPI+가 32bit COM). 기존 64bit 파이프라인과 분리 실행 |

## 사용 TR

- **opt10032 (거래대금상위요청)** — 메인 수집 TR. 입력: 시장구분(001=KOSPI/101=KOSDAQ),
  관리종목포함(1), 거래소구분(1=KRX). 출력 컬럼: 종목코드, 현재순위, 전일순위, 종목명,
  현재가, 전일대비기호, 전일대비, 등락률, 매도호가, 매수호가, 현재거래량, 전일거래량, 거래대금
- opt10081(주식일봉차트조회) — 개별 종목 보조 검증용. 전종목 루프는 호출 제한(초당 5회)
  때문에 비효율적이므로 메인 수집에는 사용하지 않음

## 환경 구성 (완료, 재현용 기록)

```powershell
# 32bit Python 3.9 설치 후
py -3.9-32 -m pip install pandas==2.0.3   # 2.x wheel 빌드 실패 회피, win32 wheel 존재 버전
py -3.9-32 -m pip install "numpy<2.0"     # pandas 2.0.3과 ABI 맞춤 (NumPy 2.x와 충돌)
py -3.9-32 -m pip install pykiwoom        # pyqt5, pywin32 함께 설치됨
```

키움 OpenAPI+ AUTO 로그인 등록 완료 (KOA Studio 및 Python 양쪽에서 무인 로그인 검증됨).
**주의**: AUTO 로그인 상태에서는 버전처리가 진행되지 않음 — 키움 업데이트 공지 시
수동 로그인 1회로 버전처리 필요.

## 구현

`src/ssen/collect/kiwoom/collect_kiwoom_money.py` (32bit 전용, `py -3.9-32`로 실행).
출력: `data/incoming/kiwoom/kiwoom_money_YYYYMMDD.csv`, SSen_Money.py(A~M열)와 동일 스키마.
ETF 제외 후 순위를 재계산하므로 기존 `순위` 컬럼 의미가 data.go.kr본과 정확히 일치하지 않음
(ETF 제외 모집단 내 순위) — `derived_columns.md`의 `기여도순위` 계산에 영향 없음
(그쪽은 날짜별 기여점수로 별도 재계산하므로 무관).

**버그 수정 기록 (2026-06-17 실거래일 통합 테스트에서 발견)**: ETF 필터링 후
`df[~df["_is_etf"]]`가 비연속 인덱스를 남긴 상태에서 `out["순위"]=range(...)`로 새
0..N-1 인덱스를 만들면, pandas가 두 인덱스를 정렬(align)하면서 종목코드/종목명 등 다른
컬럼이 NaN으로 깨짐 → `reset_index(drop=True)`를 필터링 직후에 추가해 수정.

## 코스피시세 (OPT20006 업종일봉조회요청)

`src/ssen/collect/kiwoom/collect_kiwoom_kospi.py` (32bit 전용).
출력: `data/incoming/kiwoom/kiwoom_kospi_{start}_{end}.csv`, 코스피지수_시세.py와 동일
스키마(Date/Open/High/Low/Close/Volume/Increase rate).

검증된 사실 (2026-06-17 실측, pykiwoom 실거래 호출):
- 입력: `업종코드`(001=코스피종합, 101=코스닥종합), `기준일자`(YYYYMMDD)
- **차트형 TR이라 과거 임의 날짜 백필 가능** — `기준일자=20260101` 호출 시 600행 반환 확인
  (opt10032/opt20009와 달리 "당일만" 제약이 없음 — 이게 코스피시세를 키움으로 완전히
  대체할 수 있는 핵심 이유)
- 가격 단위: 지수값×100 정수, 부호 없음 (872660 → 8726.60). OPT20009의 동일 날짜 값
  (+8726.60)과 정확히 일치 — 교차검증 완료
- 등락률/전일종가 필드가 비어있어 직접 계산 필요 → 배치 응답이 항상 충분한 과거분을
  포함하므로 연속일 종가로 `pct_change()` 산출
- **거래량 단위 = 천주(×1000 필요)** (2026-06-17 발견·수정) — data.go.kr 06-16 Volume
  =586,336,867 vs 키움 거래량=586337, 비율=999.9986 ≈ 1000. 최초 구현 시 누락되어
  06-17 수집분이 약 1/1000으로 잘못 적재됨 → `collect_kiwoom_kospi.py`에 `*1000.0` 추가,
  영향받은 날짜 재수집·덮어씀 (아래 "버그 수정 기록" 참조)

## ADR/상승하락비율 (OPT20009 업종현재가일별요청)

`src/ssen/collect/kiwoom/collect_kiwoom_adr.py` (32bit 전용, 당일 전용).
출력: `data/incoming/kiwoom/kiwoom_adr_{date}.csv`, 지수_상승종목수.py와 동일 스키마
(날짜/지수/상승종목수/하락종목수/보합종목수/하락 대비 상승비율).

검증된 사실 (2026-06-17 실측):
- 입력: `시장구분`(0=코스피/1=코스닥), `업종코드`(001/101)
- 싱글데이터에 상승/보합/하락/상한/하한/상장종목수 모두 포함 (실측: 코스피 상승349
  보합42 하락526, 코스닥 상승957 보합96 하락681)
- **과거 날짜 조회 불가** — 멀티데이터(`업종현재가`, 일별 과거치)에는 일자/현재가/등락률/
  거래량만 있고 상승·하락·보합 종목수가 없음. 키움 TR 223개(`C:\OpenAPI\koatrinputlegend.ini`
  전수 조사) 중 과거 날짜별 ADR을 제공하는 TR은 존재하지 않음 → 과거 백필은 반드시
  data.go.kr 유지, 이 스크립트는 당일 전용
- `OPT20003`(전업종지수요청)도 동일 필드(상승/보합/하락 등)를 제공하지만 마찬가지로
  당일 스냅샷만 가능 — 입력을 `업종코드`로 주면 코스피 산하 전체 세부지수(대형주/중형주/...)
  를 한 번에 반환하는 차이만 있음 (OPT20009 대체 불필요, 현재 OPT20009로 충분)

### ADR 확정 시점 검증 (진행 중)

같은 거래일에 15:35/16:30 두 번 호출해 상승/하락/보합 수치가 동일한지 확인하는
`src/ssen/collect/kiwoom/verify_adr_timing.py`를 작성하고, 다음 거래일(2026-06-18)
15:35/16:30/16:35(비교)에 실행되는 Windows 작업 스케줄러 1회성 작업
(`SSenADRVerify_1535`/`SSenADRVerify_1630`/`SSenADRVerify_Compare`)을 등록했다.
결과는 `data/adr_verify/{date}_{label}.json`과 `logs/adr_verify.log`에 기록됨.
**결과 확인 전까지 daily_update.py는 "장마감 직후 배치 실행 시점" 그대로 수집** —
동일하면 15:40 그대로, 다르면 16:00 이후로 조정 필요 (확인 후 본 문서에 결과 추가할 것).

## 개별종목 OHLCV (OPT10081 주식일봉차트조회요청)

**배경**: 기존 `src/ssen/market/collect_ohlcv.py`(data.go.kr, 전종목)가 2026-06-11
1회 수동 실행 이후 자동화가 안 돼 6일간 멈춰 RS_breakout_strategy.py의 RS 계산이
전부 깨지는 사고가 발생 (원인: 에러가 아니라 daily_update.py/scheduler 어디에도
연결되지 않은 완전 독립 수동 스크립트였음 — `docs/perf_report_*` 류가 아니라
운영 프로세스 문제). 이를 계기로 OHLCV도 키움으로 일부 전환.

`src/ssen/collect/kiwoom/collect_kiwoom_ohlcv.py` (32bit 전용).

검증된 사실 (2026-06-17 실측, 원익IPS 240810):
- 1회 호출 소요시간 **0.45초**, 600행(약 2.4년치) 반환 — opt20006과 같은 차트형 TR이라
  **종목당 1회 호출로 당일+백필 모두 충분**
- 거래대금 단위 = **백만원** (×1,000,000, opt10032와 동일 패턴 — 06-12 거래대금×100만
  =690.95억 vs data.go.kr 같은 날 거래대금 690.95억대로 교차검증)
- 거래량/가격은 raw 단위 (data.go.kr과 동일, 변환 불필요)
- 응답에 종목코드는 0행만 채워지고 나머지 공란, **종목명/시장구분 필드 자체가 없음**
  → `dim_stock` 테이블에서 code→market join 필수 (daily_update.py의
  `merge_ohlcv_into_market_store()`에서 처리)
- 호출 제한: 키움 개발가이드(`koa_devguide.xml`) 1차 출처 확인 — **"데이터 조회는
  1초당 5회로 제한"** (CommRqData 포함) + 수치 비공개 "서버부하방지 제한" 별도 존재
  → 안전마진으로 호출 간 0.3초 슬립 적용(약 3.3회/초 운영)

**대상 범위(universe) 결정 — 전종목 대신 거래대금 상위 1년 누적**:
전종목(KOSPI+KOSDAQ 약 2,500개) 매일 수집은 호출시간이 부담스러워(약 19분/일),
`fact_daily_stock`에서 최근 250거래일(`RS_PERIOD_12M=250`과 정의 일치) 동안 한 번이라도
거래대금 상위에 든 종목만 매일 키움으로 갱신. "상위 300종목/일" 단순안은 RS 백분위
모집단을 왜곡시켜 기각됨(RS는 전체 종목 대비 percentile rank이므로 모집단이 좁아지면
의미가 달라짐).

실측 universe 크기: **1,242종목** (최초 추정 743종목보다 큼 — 250거래일 lookback이
2025-06-10까지 걸쳐 그만큼 더 많은 종목이 rotation에 포함됨).

| 항목 | 값 |
|---|---|
| universe 크기 | 1,242종목 |
| 예상 소요시간 (0.45초/call + 0.3초 슬립) | 약 15.5분 |
| **실측 소요시간** (2026-06-17 전체 1회차) | **약 13.5분** (1,236/1,242종목 성공, 6종목 실패) |
| 수집 결과 | 318,247행 |

**실행 중 발견·수정한 추가 버그**: `merge_ohlcv_into_market_store()`가 종목코드→시장구분
매핑을 `dim_stock`(Postgres)에서 가져왔는데, 이 테이블은 최근 6개월치만 유지되어
250거래일 universe의 상당수가 매핑 실패 — 1차 실행에서 318,247행 중 **84,206행
(26%)이 스킵**됨. `fact_daily_stock` parquet(같은 250거래일 lookback)에서 직접
매핑하도록 수정해 전량 반영됨 (`_load_code_market_map()`).

**부수적으로 발견한 기존 데이터 품질 이슈 (재진단 완료, 2026-06-17)**: `data/market/ohlcv`
저장소에 3개 종목(034230 파라다이스/066970 엘앤에프/090460 비에이치)이 KOSPI/KOSDAQ
양쪽 폴더에 동시 존재해 날짜+종목코드 기준 804행 중복, RS의 `pivot()`이 "Index
contains duplicate entries"로 실패.

최초엔 "data.go.kr이 mrktCls 양쪽으로 동시 반환"으로 추정했으나 **재조회로 반증**:
- data.go.kr `getStockPriceInfo` 원본 응답 필드 직접 확인 결과 시장구분(`mrktCtg`)
  단일 필드만 존재, 거래소구분(KRX/NXT) 필드 자체가 없음 — "필드 혼동 매핑"이 아니라
  애초에 그 구분이 데이터에 없음
- 위 3종목을 `mrktCls=KOSPI`/`mrktCls=KOSDAQ`로 각각 라이브 재질의(2026-06-01~17)
  한 결과 **KOSPI 33건 / KOSDAQ 0건** — 현재 data.go.kr은 KOSPI 단일 분류로만 반환,
  동시반환 현상이 라이브 데이터에 없음
- 키움 OPT10081도 입력에 거래소구분이 없고(종목코드/기준일자/수정주가구분뿐),
  NXT/통합 데이터는 종목코드에 `_NX`/`_AL` 접미사를 붙여야만 나오는데 본 스크립트는
  항상 순수 6자리 코드만 사용 → NXT 혼입 메커니즘 자체가 없음 (전체 저장소 재스캔
  결과 이 3종목 외 시장 불일치 0건 확인)
- **결론**: 거래소(KRX/NXT) 동시상장이 아니라, 해당 종목들의 과거 시장 이전상장
  이력 등으로 생긴 **stale 잔존 데이터**로 판정 (data.go.kr에 거래소구분 필드가
  없어 KRX/NXT 기준 필터링은 적용 불가능)
- **가드 정책**: "종목 1개 = `fact_daily_stock` 최신 시장구분 기준 단일 폴더만 유지"
  — `merge_ohlcv_into_market_store()`에 `_purge_stale_cross_market_rows()` 추가,
  매 merge 시 방금 쓴 market과 다른 폴더에 같은 종목코드가 남아있으면 자동 제거

**하이브리드 구조**: universe 종목 → 키움(당일 즉시, `data/market/ohlcv`에 upsert),
universe 밖 종목 → 기존 `collect_ohlcv.py`(data.go.kr, 전종목)가 daily_update.py
실행 시마다 함께 호출되어 1~2일 지연으로 보완 (`run_ohlcv_datagokr_backstop()`).
universe든 아니든 동일한 `data/market/ohlcv/yearmonth=YYYYMM/{market}/data.parquet`
저장소에 날짜+종목코드 키로 upsert되므로 strategy 코드(`supertrend_strategy.py` 등)는
소스 구분 없이 그대로 사용.

## 버그 기록: 가격 부호 오염 + OHLCV name 덮어쓰기 회귀 (2026-06-17, 기간분석/트레이드 탭)

**버그 1 — 키움 가격 필드 부호 오염으로 테마 누적수익률 -100% 미만 표시**

증상: 기간분석 탭 하락 테마 TOP10이 -103.5%~-212.0%로 표시(복리/평균 산식 모두
이론상 -100%를 넘을 수 없음). 원인은 테마 dedup 로직이 아니라 **`collect_kiwoom_money.py`의
가격 부호 처리**: 키움 OPT10032의 `현재가` 필드는 등락 방향을 나타내는 +/- 기호를
붙이는데(대비/등락률처럼), 가격 자체는 절대 음수가 될 수 없음에도 `parse_signed_number`를
그대로 적용해 음수로 저장됨 (실측: 현대로템 064350 06-17 종료일종가 -219,000원,
시작일기준가 -211,500원). 시가총액=상장주식수×종료일종가도 음수가 되어 06-17 거래대금
상위 60개 하락종목 전체에 영향. `parse_price()`(절댓값 전용 파서) 신설로 수정,
06-17 데이터 키움 재수집 + 전체 파이프라인 재처리로 파생 컬럼(기여점수/규모 등)까지
재계산함. 보정 후 방산(064350 단일종목) 1개월 누적수익률 -212.0% → **+12.0%**로 정상화,
falling TOP10 전체 -48.35%~-10.18%로 이론적 범위 안에 들어옴 확인.

**버그 2 — OHLCV merge가 종목명을 None으로 덮어쓰는 회귀**

증상: 트레이드 탭(슈퍼트렌드/RS돌파/눌림목) 종목명이 전부 None. 원인은
`merge_ohlcv_into_market_store()`가 opt10081에 없는 `name`/`change`/`change_pct`/
`shares`/`mktcap` 컬럼을 무조건 `None`으로 채워 upsert했고, `drop_duplicates(keep='last')`
에서 키움(새 행, name=None)이 기존 data.go.kr 데이터(name=실값)를 통째로 덮어씀 —
universe OHLCV 1회 수집(기본 lookback 400일)이 **268일치 318,247행(전체의 33.4%)**의
종목명을 손상시킴. `_load_code_name_map()`(fact_daily_stock 기반)으로 name을 항상
채우고, change/change_pct/shares/mktcap은 기존 행 값을 carry-forward하도록 수정.
이미 손상된 318,247행은 같은 매핑으로 일괄 복구(0건 NULL 확인). supertrend/rs-breakout/
pullback 3개 엔드포인트 전부 재검증, name=None 0건 확인.

두 버그 모두 **키움 전환 작업에서 새로 도입된 회귀**였음 — 신규 수집 경로 추가 시
기존 데이터를 덮어쓰는 upsert 로직은 컬럼별 보존/병합 정책을 명확히 설계해야 함.

## 미해결 / 다음 확인 사항

1. **장중 변동 vs 마감 확정치 차이**: opt10032는 호출 시점까지의 장중 누적 거래대금을
   반환. 15:40 호출값이 익일 data.go.kr 확정치와 정확히 일치하는지 1회 더 대조 필요
   (`동일 날짜 종가/거래대금 cross-check` — VERIFY 단계에서 수행)
2. **opt10032 1회 호출이 거래대금 상위 몇 종목까지 주는지 상한 확인** (100개 확인됨,
   `next` 파라미터로 연속조회 시 더 받을 수 있는지 테스트 필요 — TOP_N=300 설정은
   미검증 가정이므로 첫 실행 시 VERIFY에서 실제 반환 행수 확인)
3. **시가총액(시작일기준가 역산값)의 정확도**: opt10032에 전일종가가 없어 `현재가-전일대비`로
   역산함. data.go.kr의 시작일기준가(기간 시작일 종가)와는 정의가 다르므로
   `calc_derived.py`의 O열(코스피 대비 등락률) 계산 시 혼용 금지 — 단일 거래일 수집
   구조이므로 시작일=종료일이라 정의상 문제 없음
4. **키움 OpenAPI 약관상 데이터 재배포 제한** — SSenBoard를 상용 서비스로 전환 시
   키움증권에 사용 범위(개인 분석 vs 제3자 제공) 문의 필요 (법적 검토 사항, 코드 작업 아님)
