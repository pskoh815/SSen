# fact_kospi Volume 단위 오류 보정 (2026-06-17)

## 배경

키움 OPT20006 코스피시세 전환 작업 중, 06-17(키움) Volume이 06-15(data.go.kr)
대비 약 900배 작아 보이는 현상을 발견. 인베스팅닷컴 코스피지수 과거 데이터
(2020-01-02~2026-06-17, 1586행, 사용자 제공 `코스피지수 과거 데이터 2020년-2026년.csv`)로
전구간을 교차검증함.

## 진단 결과

### 1. 날짜별 대조 (2020-01-02~2026-06-09, 1579일 매칭)

| 구분 | 건수 | 비율 |
|---|---|---|
| 비율 ≈1000 (±5%) | 1570 | **99.43%** |
| 비율 ≠1000 | 9 | 0.57% |

비율≠1000인 9건은 모두 다른 이유로 설명됨 — 새로운 버그 아님:
- 3건(2022-03-30, 2022-04-25, 2022-04-29): `fact_kospi`에 해당 날짜 행 자체가 결측
  (Volume 단위 문제와 무관한 별개 데이터 누락 — 이번 작업 범위 제외, SUMMARY 참조)
- 6건(2026-06-01, 02, 04, 05, 08, 09): 사용자가 이전에 수동으로 직접 고친 날짜라
  이미 정상 단위(비율≈1.0)

→ 마스터 구간에 속하는 1570개 날짜 **100%가 정확히 ×1000 패턴**. 우연이 아닌 전구간
일관된 결함으로 판정.

### 2. 원인 — 코드 버그 아닌 원본 마스터 xlsx 결함

`data/archive/pre_existing_20260610/SSen분석.xlsx`의 "코스피시세" 시트를 직접 열어 확인:

```
2020-01-02   Volume=494,680      ← 엑셀 원본에 이미 이 값 (정상치는 약 4억9천만)
2026-06-01   Volume=636,175,294  ← 같은 파일 뒤쪽(사용자가 수동 수정한 구간)은 이미 정상값
```

- `excel_to_parquet.py`(`_read_fact_kospi`)는 `pd.to_numeric`만 적용, 배율 변환 로직 없음
- `calc_derived.py`는 코스피 Volume을 전혀 참조하지 않음 (Close만 사용)
- **결론: ingest 코드 버그가 아니라 원본 SSen분석.xlsx 마스터 파일 자체의 결함.**
  2026-06-01 이후 구간만 사용자가 수동으로 직접 고쳐 정상값이 들어가 있었고,
  2020-01-02~2026-06-09(2026-06-01~09 제외) 구간은 그대로 남아 있었음.

### 3. 영향 범위 분석 — 백테스트/신호 영향 없음

코드 전체 검색(grep) 결과:
- **BullBear 매크로 필터(BR5/McClellan/A-D Line)**: `fact_adr`(상승/하락 종목수)만 사용,
  `fact_kospi`를 참조하지 않음 → 영향 없음
- **kkangto_spec.md 매매시스템**: "거래량" 언급은 모두 종목별(`fact_daily_stock`)
  베이스돌파/클라이맥스 판정 — 코스피 지수 Volume과 무관 → 영향 없음
- **`fact_kospi.volume`이 실제로 쓰이는 곳**: `period_analysis.py`의 `_load_kospi()`는
  `date, close`만 SELECT — Volume 컬럼은 코드 어디서도 소비되지 않는 순수 표시용 컬럼
- → **6년치 버그가 백테스트 신호·필터 결과에 어떤 영향도 주지 않았음을 확인.**

## 보정 작업

1. `fact_kospi` 2020-01-02~2026-06-09 구간 Volume에 ×1,000 일괄 적용
   (Open/High/Low/Close/change_rate 컬럼은 손대지 않음)
2. **작업 중 발견·즉시 수정한 실수**: 최초 일괄 적용 시 사용자가 이미 수동으로 고쳤던
   2026-06-01/02/04/05/08/09(6일)까지 범위에 포함되어 중복으로 ×1000이 한 번 더
   적용되는 사고가 발생(해당 6일 Volume이 일시적으로 6,000억대까지 치솟음). VERIFY
   단계에서 describe() 평균이 26억대로 비정상적으로 높게 나와 즉시 발견, 해당 6일만
   ÷1,000으로 되돌려 정정함. 최종 데이터에는 영향 없음(정정 후 검증으로 확인).
3. `data/parquet/_manifest.json`(catalog.json) 체크섬·rowcount 78개 파티션 전체 갱신
   (`_write_partition()` 재사용, 영향받은 파티션만 재계산)
4. Postgres `fact_kospi_index` 78개월 파티션 전체 재적재 (`load_parquet_to_postgres.load()`)
5. **ingest 단계 sanity check 추가** (`src/ssen/etl/convert_excel_to_parquet.py`
   `_read_fact_kospi()`): 코스피 Volume이 100만 미만인 행이 있으면 `[경고]` 로그 출력
   (코스피 전체 거래량은 항상 수억대이므로, 100만 미만은 단위 누락 의심 — 마스터 xlsx가
   재입수되어도 동일 오류가 조용히 재발하지 않도록 하는 최소 방어선)
