# OHLCV 백필 2015-01-01 ~ 2019-12-31

## 배경

RS/슈퍼트렌드/눌림목/RS돌파 전략(`data/market/ohlcv`)이 2015~2019년 구간을 조회할 때
lookback 부족(250거래일)으로 계산이 깨지는 문제를 보완하기 위해, 키움 OPT10081
(수정주가구분="1")로 universe(거래대금 상위 250거래일 누적, 1,247종목) 전체의
2015~2019년 일봉을 백필했다.

## PLAN

- universe: `compute_ohlcv_universe()` 기준 1,247종목 (기존 daily_update.py와 동일 정의)
- OPT10081 1회 호출은 ~600거래일(약 2.4년)만 반환 → 5년을 끊김없이 덮기 위해
  기준일자를 2016-12-31 / 2018-12-31 / 2019-12-31로 잡아 3패스 진행 (구간 중첩 설계)
- 안전장치: `collect_kiwoom_ohlcv.py` 기존 증분저장(20종목마다)/워치독(30초)/`--resume` 재사용
- 예상 소요: 종목당 ~0.75초(호출+슬립) × 1,247종목 × 3패스 ≈ 35~60분

## 발견된 문제: 패스 경계 가격단절 (43건)

전체 스캔(인접 등장일 가격비율 4배 이상/0.25배 이하, 간격 10일 이내) 결과 **43건**
발견. 패턴 분석 결과 신성이엔지(011930, 6/19 별도 처리)류의 "실제 병합/분할 미반영"이
아니라, **키움 OPT10081의 수정주가가 호출 시점의 `기준일자`까지 알려진 이벤트만
반영**하는 것이 원인으로 추정됨:

- Pass1(기준일자=2016-12-31)은 삼성전자의 2018-05-04 50:1 분할을 "아직 모르는 시점"
  기준이라 미반영가(1,533,000원대)를 반환
- Pass2(기준일자=2018-12-31)는 분할 이후 시점이라 같은 2016년 데이터도 분할반영가
  (30,660원대)로 반환
- 두 패스를 단순 upsert(겹치는 날짜는 나중 패스 값으로 덮어쓰기)하다 보니, 겹치지
  않는 경계 정확히 하루(Pass1 단독 구간의 마지막 날 ↔ Pass2 단독 구간의 첫 날)에서
  기준이 다른 두 시리즈가 이어붙어 단절 발생

43건 모두 경계일이 정확히 두 지점(Pass1↔2: 2016-07-18→19, Pass2↔3: 2017-07-17→18)에
집중되어 있어 이 가설을 강하게 뒷받침함.

## 보정 적용

각 종목의 경계일 비율(= 경계 이후 종가 / 경계 이전 종가)을 그 종목의 경계 이전 전체
구간(open/high/low/close)에 곱해 재정렬. 거래량/거래대금은 변경하지 않음(가격 연속성만
보정 대상).

- 비율은 원칙적으로 **거래량>0인 가장 가까운 인접일** 기준으로 계산(거래정지/미체결일의
  고시가는 신뢰도가 낮음)
- 예외: 042660, 066430은 거래정지가 1년 이상 지속되어 인접일 대체 시 실제 시장 재평가가
  섞여 들어가는 부작용이 더 커서, 경계일 원값(고시가) 그대로 사용

### 버그 발견 및 수정 (작업 중)

인접일 대체를 사용한 3종목(217270/234100/042000)에서, 비율 계산에 쓴 인접일과 보정을
적용해야 할 실제 경계일(예: 2016-07-18) 사이 구간이 보정 범위(cutoff)에서 누락되는
버그 발견 — 비율 계산용 "인접일"과 보정 적용 범위인 "실제 경계일"을 혼동한 것이 원인.
재스캔(`count: 6`)으로 발견 즉시 해당 구간(3종목, 59행)만 추가 보정.
